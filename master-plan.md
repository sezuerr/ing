## 2. Product — Lightweight PRD

### 2.1 Problem and User

**Problem:** College students want to form offline friendships with peers who share their interests, but strong personal social boundaries, heavy academic workloads, and a lack of natural in-person interaction channels prevent these connections from forming.

**User:** A college student (undergraduate or graduate) who wants to meet like-minded peers on campus. They are open to new friendships but find approaching strangers directly uncomfortable. They prefer low-pressure, interest-anchored interactions over forced socialising.

### 2.2 Product Goal

This prototype demonstrates that a college student can discover and connect with a peer who shares their interests by anonymously posting, mutually liking, and chatting — all without revealing their identity until both parties opt in.

### 2.3 Core Demo Flow

1. User opens the mini program and lands on the Discover feed — a scrollable list of anonymous posts, each with text and interest tags.
2. User taps "+" to create their own post: they write a short message (e.g., "Looking for a gym buddy, I go to the campus gym Tues/Thurs evenings") and select 1–3 interest tags (e.g., "健身", "自律").
3. The post appears in the feed. Another user sees it, finds it relatable, and taps the like (heart) icon.
4. The post author receives an in-app match notification: "Someone liked your post — and you liked theirs too!" (mutual like detected).
5. Both users are taken to a chat screen where they can exchange text messages.
6. Users continue chatting and decide to meet offline — the core journey is complete.

### 2.4 Must-Haves

- [ ] Anonymous post creation with text body and interest tags
- [ ] Scrollable feed of posts from other users
- [ ] Like action on posts with mutual-like detection
- [ ] Match creation when two users mutually like each other's posts
- [ ] Chat screen with real-time message sending and receiving
- [ ] Match notification informing users when a match occurs

### 2.5 Non-Goals

- No user accounts, profile photos, or real-name registration — anonymous by design
- No image or media uploads in posts — text and tags only for the walking skeleton
- No friend list, contact book, or social graph browsing
- No push notifications (in-app only for the walking skeleton)
- No content moderation or reporting system — deferred to post-prototype
- No group chat or multi-person conversations
- No algorithmic feed ranking — chronological sort only
- No location-based discovery beyond campus affiliation
- No AI-powered matching or recommendation

### 2.6 Walking Skeleton

> A user opens the mini program, sees the Discover feed populated with anonymous posts. They create a post with text and interest tags. Another user likes that post. The system detects a mutual like (the author had previously liked one of the other user's posts) and creates a match. Both users receive a match notification and are taken to a 1-on-1 chat screen where they can send and receive text messages. The full journey — from anonymous browsing to direct conversation — is traversable end-to-end with cloud-persisted data.

### 2.7 Prototype Success Evidence

- A user unfamiliar with the product can complete the core demo flow (browse → post → match → chat) without guidance in under 3 minutes
- Mutual-like match detection fires correctly: a match is created only when both users have liked each other's posts, never on a single-sided like
- Chat messages persist and are delivered to the correct conversation between the two matched users
- The anonymous boundary is preserved: at no point before a mutual match can a user see who wrote a post or initiate a direct message

## 3. UX

### 3.1 Experience Principles

1. **Anonymous-first:** At every touchpoint, the user should feel their identity is protected. No real names, no profile photos, no contact information is ever exposed before a mutual match. Post authorship is never attributable.
2. **Low-pressure:** Every interaction is opt-in. Liking a post is a single tap with no required follow-up. A match feels like a pleasant surprise, not a commitment.
3. **Interest-anchored:** Posts are organised around what people want to do together, not who they are. Interest tags are visually prominent and serve as the primary discovery mechanism.
4. **College-appropriate:** The tone is warm, casual, and peer-to-peer — never dating-app-coded, never corporate. The design language should feel like a campus bulletin board, not a social network.

### 3.2 Design Language

- **Visual tone:** Clean, warm, and approachable — soft rounded corners, generous whitespace, one accent colour on a neutral background
- **Typography:** System font (WeChat's default sans-serif), high readability at 14–16px body size
- **Colour:** Neutral base (white/grey backgrounds) with one warm accent colour (e.g., coral or warm orange) for actions and highlights
- **Iconography:** Simple line icons, consistent stroke weight, platform-standard where possible
- **Reference:** No specific reference — the design should feel native to WeChat mini programs

### 3.3 Interaction Guidelines

- Every async operation (post creation, like, message send) shows a brief loading indicator
- Destructive actions (deleting own post) require confirmation
- Empty states include a prompt: "还没有帖子，成为第一个发声的人吧" for an empty feed
- Error states show a friendly message with a retry action where applicable
- Pull-to-refresh on the feed; pull-up to load more posts
- Chat input area is fixed at the bottom of the screen, mirroring WeChat's native chat UX

### 3.4 Screen Inventory

| Screen | Purpose | Notes |
|--------|---------|-------|
| Discover Feed | Browse anonymous posts; like posts; pull to refresh | Main entry screen. Chronological list of post cards. |
| Create Post | Compose a new anonymous post with text and interest tags | Modal or dedicated page. Tag selection from predefined list. |
| My Posts | View own posts and their like counts | Accessible from Discover or a tab. |
| Match Notification | In-app alert when a mutual like occurs | Could be a modal or a dedicated transition screen. |
| Chat | 1-on-1 text messaging with a matched user | Message list + fixed bottom input. |
| Conversation List | Overview of all active chats | Accessible from a "Messages" tab or entry point. |

### 3.5 Primary Journey

| Step | Screen | User Action | System Response |
|------|--------|-------------|-----------------|
| 1 | Discover Feed | Opens the mini program | Feed loads with recent anonymous posts from cloud DB |
| 2 | Discover Feed | Scrolls through posts, reads a few | — |
| 3 | Create Post | Taps "+" button, writes text, selects tags, taps "发布" | Post is saved to cloud DB, appears at top of feed |
| 4 | Discover Feed | Taps the like (heart) icon on a post they find interesting | Like is recorded in cloud DB; system checks for mutual like — no match yet |
| 5 | Match Notification | (Later) Receives a match alert — another user liked their post and the system detected a mutual like | Match record created; conversation opened; both users notified |
| 6 | Chat | Types a message, taps send | Message saved to cloud DB and delivered to the other user's chat screen |
| 7 | Chat | Receives a reply from the matched user | New message appears in the chat; conversation continues |

### 3.6 Key States

**Discover Feed**
- Default / loaded state: List of post cards, each showing anonymous avatar placeholder, text preview, interest tags, like count, and relative timestamp
- Loading state: Skeleton cards (grey placeholder rectangles) while initial fetch is in progress
- Empty state: Illustration + "还没有帖子" + "发布第一条" call-to-action button
- Error state: "加载失败" message + "重试" button
- Pull-to-refresh: Spinner at top of list
- End-of-list: Subtle "没有更多了" divider

**Create Post**
- Default state: Empty text area with placeholder "分享你想做的事...", tag selector below with tappable chips
- Validation state: "发布" button disabled until text ≥ 1 character and at least 1 tag selected
- Submitting state: Button shows spinner, "发布中..."
- Error state: Toast "发布失败，请重试", form remains filled
- Success state: Redirect back to Discover Feed, new post visible at top

**Chat**
- Default state: Message list (empty for new conversations), fixed input bar at bottom
- Loading state: Spinner while fetching message history
- Sending state: Message appears immediately in the list with a pending/sending indicator
- Send error state: Failed message shows retry icon; tap to resend
- Empty state (new conversation): "发送第一条消息，打个招呼吧" prompt

**Match Notification**
- Trigger state: Overlay or modal: "🎉 你们互相点赞了！" with the matched post preview and "开始聊天" button
- Dismissed state: User is taken to the Chat screen or returns to previous screen

## 4. Architecture — ADR

### 4.1 Prototype Pattern

> **[RECOMMENDATION]** Pattern: Thin client + cloud API (微信云开发全栈)

The WeChat Mini Program serves as the thin client — rendering UI, capturing user input, and managing local state. All shared data (posts, likes, matches, messages) is persisted in 微信云开发 (cloud DB + cloud functions). There is no separate application server. Cloud functions handle the match-detection logic (checking for mutual likes and creating conversations) because this involves multi-collection writes that are best executed server-side for consistency. This pattern is the natural fit for the single approved pathway and requires no additional infrastructure beyond what 微信云开发 provides out of the box.

### 4.2 Selected Technical Pathway

> **Pathway selected:** 微信小程序 + 微信云开发
> **Selected from:** [Single pathway supplied]

**Rationale:** This is the only approved pathway and it is well-suited to the brief. 微信云开发 provides cloud database (JSON-document store), cloud functions (Node.js serverless compute), and cloud storage — all within the WeChat ecosystem and accessible via `wx.cloud` APIs without separate authentication. The existing project already uses this pathway with cloud environment `cloud1-d3g3i3b996dd7dad5` and has deployed collections for users, posts, likes, matches, conversations, and messages — the data infrastructure for the walking skeleton is already in place.

**Trade-offs accepted:**
- Vendor lock-in to the WeChat ecosystem — acceptable because the brief targets WeChat users exclusively
- Cloud function cold-start latency — acceptable for a prototype where sub-second response times are not required
- 云开发 database lacks relational joins — acceptable because the data model (posts, likes, matches, messages) is document-oriented by nature
- Real-time chat via database watchers or polling rather than WebSocket — acceptable because instant delivery is not a core requirement for the walking skeleton

**Outside the boundary:** Content moderation service (deferred), push notification service (deferred via WeChat 订阅消息 — mocked for prototype), analytics, any third-party identity verification.

### 4.4 Data Model and Storage

| Entity | Key Fields | Notes |
|--------|-----------|-------|
| Post | `_id`, `_openid` (author), `content` (text, ≤500 chars), `tags` (array of tag IDs), `like_count`, `created_at` | Anonymous authorship via openId — author identity is never surfaced to other users |
| User | `_openid`, `nickname` (system-generated alias e.g. "同学#1234"), `campus` | Minimal profile. openId is the stable identifier; nickname is auto-generated for display |
| Like | `_id`, `_openid` (who liked), `post_id`, `created_at` | One like per user per post (unique constraint on `_openid + post_id`) |
| Match | `_id`, `user_a_openid`, `user_b_openid`, `post_a_id`, `post_b_id`, `created_at` | Created when mutual like detected. Stores which posts triggered the match. |
| Conversation | `_id`, `match_id`, `participants` (array of two openIds), `last_message_preview`, `updated_at` | One conversation per match. Used for the conversation list screen. |
| Message | `_id`, `conversation_id`, `_openid` (sender), `text`, `created_at` | Text-only for walking skeleton. Ordered by `created_at` ascending in chat view. |

**Storage approach:** > [RECOMMENDATION] 微信云开发 cloud database (JSON-document store). All collections live in cloud environment `cloud1-d3g3i3b996dd7dad5`. Data access is through cloud functions (`api`) which run with admin privileges — the client never accesses the database directly, ensuring collection-level permissions remain restrictive.

**Rationale:** Cloud database is the native persistence layer for 微信云开发. No additional database setup, no connection strings, no separate hosting. The admin-privilege cloud function pattern is already established in the existing codebase (`utils/cloud.js` routes all data access through `wx.cloud.callFunction`).

**Sensitive data:** openId serves as the user identifier but is never displayed to other users. Post content is user-generated text — content moderation is deferred per non-goals. No real names, no phone numbers, no email addresses, no location data beyond optional campus affiliation.

### 4.5 AI Boundary

No AI role in this prototype.

The core mechanism — mutual-like matching — is a deterministic rule (if user A liked post by user B AND user B liked post by user A → create match). No recommendation algorithm, content understanding, or generative AI is required for the walking skeleton.

### 4.6 Mocked Components

| Component | What It Replaces | Mock Behaviour | Future Replacement |
|-----------|-----------------|---------------|-------------------|
| Content moderation | Automated text/image scanning for policy violations (e.g., 微信云开发 content security API) | All posts pass moderation automatically; no filtering in prototype | Integrate `wx.cloud.callFunction` to call 微信内容安全 API before post creation; reject or flag violating content |
| Push notifications | 微信订阅消息 for notifying users of new matches when the app is closed | In-app match notification only (modal/toast while the mini program is open). No out-of-app alerts. | Implement 订阅消息 templates for match notifications; user opts in to receive them |
| Interest tag taxonomy | Curated, managed list of interest categories with localisation | Static JSON seed file with 15–20 common campus interest tags (健身, 学习, 美食, 游戏, 音乐, 摄影, 运动, 读书, 电影, 旅行, 编程, 艺术, 舞蹈, 志愿者, 创业) | Move tags to a cloud database collection for dynamic management; add admin UI for tag curation |
| Campus affiliation | University verification or .edu email validation | User selects campus from a static list (or the field is optional). No verification. | Integrate with university SSO, geofencing, or manual approval if campus restriction becomes a requirement |

### 4.7 Privacy and Security Constraints

- No real user data (names, phone numbers, emails, student IDs) is collected or stored in this prototype
- WeChat openId is used as the anonymous identifier — it is never displayed to other users or exposed in post/chat UIs
- All database access is mediated through cloud functions with admin privileges — the client has no direct collection read/write permissions
- Post content is the only user-generated content exposed to other users; it is text-only (no images, no files)
- Cloud environment ID and any API keys must not be hardcoded in client-side code — use `wx.cloud.init()` with the environment ID in a config
- 云开发 collection permissions should remain at their restrictive defaults ("仅创建者可读写") since cloud functions bypass collection-level ACLs

### 4.8 Alternatives Considered

| Alternative | Why Not Selected |
|-------------|----------------|
| Client-side only (纯前端 + 本地 mock 数据) | The core journey requires shared data: posts must be visible to other users, likes must be tracked across users, matches must be detected across users, and chat messages must be exchanged. Client-side only cannot demonstrate the multi-user flow. |
| Separate backend server (e.g., Node.js + MySQL on a VPS) | Adds infrastructure complexity (server provisioning, database admin, API design) that 微信云开发 handles out of the box. No benefit for this prototype. |
| WebSocket for real-time chat | Requires additional infrastructure (WebSocket server, connection management). 云开发 database watchers or polling achieve "real-time enough" for the walking skeleton with zero additional setup. |
| Full user profiles with photos and bios | The brief's core insight is that anonymity lowers the barrier to connection. Rich profiles work against this principle and add moderation/safety complexity. |

### 4.9 Rationale Summary

The walking skeleton requires shared, persistent data across multiple users — anonymous posts, likes, match detection, and chat messages. 微信云开发 provides this with zero additional infrastructure: cloud database for persistence, cloud functions for match-detection logic, and the existing `api` cloud function pattern from the codebase. The thin-client pattern keeps the Mini Program focused on UI and state, while admin-privilege cloud functions handle all data access securely. Every component not essential to the core demo flow (moderation, push notifications, tag management) is mocked or deferred, keeping the skeleton lean and buildable within 1–2 weeks.

## 5. Milestones

> Each slice takes approximately 1–3 days. Maximum 6 slices. Each slice is independently demonstrable.

### Slice 1: Anonymous Post Feed — Walking Skeleton Foundation
**Goal:** A user can see a feed of anonymous posts and create their own post. Posts persist and are visible to all users.

**Scope:**
- Discover Feed screen with scrollable post list (chronological order)
- Post card component: anonymous author alias, text content, interest tags, relative timestamp, like button (non-functional yet)
- Create Post screen: text input (≤500 chars) + tag selector (static seed list of 15 tags)
- Cloud function `api` routes for: `posts.list` (read all), `posts.create` (write new post)
- Posts collection in cloud database, seeded with 10–15 sample posts

**Done when:** User A creates a post and User B can see it in their feed after pull-to-refresh. Feed shows posts from all users in reverse chronological order.

### Slice 2: Like + Match Detection
**Goal:** Users can like posts, and the system detects and notifies mutual likes.

**Scope:**
- Like button on post cards (heart icon, toggles filled/outline, shows count)
- Cloud function routes: `likes.toggle` (like/unlike a post), match detection logic (check if mutual like exists after each like)
- Match Notification screen (modal or inline alert: "你们互相点赞了！")
- Conversation auto-creation when a match is detected
- Conversation List screen (accessible from a tab bar "消息" entry)

**Done when:** User A likes User B's post, then User B likes User A's post → match notification appears for both users → a conversation is created and visible in the Conversation List.

### Slice 3: Chat
**Goal:** Matched users can exchange text messages in real time.

**Scope:**
- Chat screen: scrollable message list + fixed bottom input bar
- Cloud function routes: `messages.send`, `messages.list` (for a conversation)
- Message delivery: polling at 30s intervals or 云开发 database watcher for new messages in the active conversation
- Unread indicator on the Conversation List (count of unread messages per conversation)

**Done when:** Two matched users can type messages, send them, and see each other's messages appear in the chat screen within 30 seconds. Messages persist and are retrievable when reopening the conversation.

### Slice 4: Interest Discovery + My Posts
**Goal:** Users can filter the feed by interest tags and manage their own posts.

**Scope:**
- Tag filter chips at the top of Discover Feed (tap to filter by one tag, tap again to clear)
- "My Posts" section: view own posts, see like counts, delete own posts
- Post list supports pagination (load 20 at a time, "load more" at bottom)
- Empty states for filtered feed ("这个标签下还没有帖子")

**Done when:** User filters feed to show only "健身" posts, sees only relevant posts, can create a post with that tag and see it in the filtered view. User can view and delete their own posts.

### Slice 5: Safety Essentials
**Goal:** Basic safety mechanisms are in place before wider testing.

**Scope:**
- Post content validation (reject empty posts, posts over 500 chars, posts with no tags)
- Rate limiting: max 10 posts per user per day (cloud function check)
- Basic profanity filter on post content (client-side keyword blocklist, replace with ***)
- "Unmatch" action: user can leave a conversation, which hides it and prevents further messages

**Done when:** A post with banned keywords is rejected with a friendly message. A user who exceeds the daily post limit sees a rate-limit message. A user can unmatch and the conversation is archived.

### Slice 6: Polish + Demo Readiness
**Goal:** The prototype is smooth enough to demonstrate and test with users.

**Scope:**
- Loading skeletons on all screens (replace spinners with placeholder shapes)
- Pull-to-refresh on Discover Feed and Conversation List
- Haptic feedback on like action (`wx.vibrateShort()`)
- Smooth transitions: feed → post detail, match notification → chat
- Error handling pass: network failures show friendly toasts with retry, cloud function timeouts handled gracefully
- Seed data refresh: 30+ diverse posts across all interest tags for a lively demo feed

**Done when:** A user unfamiliar with the product completes the full flow (browse → filter → post → like → match → chat) without encountering a crash, a blank screen, or an unclear error state.

## 6. Handoff

### 6.1 Decisions That Should Remain Fixed

- **Technical pathway:** 微信小程序 + 微信云开发 — the entire data access pattern (cloud functions with admin privileges, cloud database collections) is built around this. Changing pathway would require rebuilding the backend layer from scratch.
- **Anonymous-first design:** The core product insight — anonymity lowers the barrier to connection — is embedded in the data model (openId never surfaced) and every screen. Adding real-name profiles or identity verification later would be a product decision that changes the fundamental user experience, not just a technical change.
- **Walking skeleton scope:** Anonymous posts → mutual like → match → chat. Adding features (images, video, group chat, algorithmic feed) before this skeleton is validated with real users risks building the wrong thing. Validate the core mechanism first.

### 6.2 Questions for the Next SDD Phase

- [ ] What is the exact interest tag taxonomy? How are tags created, managed, and potentially user-submitted?
- [ ] Should posts have an expiration/auto-archive? (e.g., posts older than 7 days are hidden)
- [ ] What is the unmatch flow exactly? Should it be reversible? Should chat history be deleted or archived?
- [ ] What are the exact rate-limit thresholds? (posts per day, likes per minute, messages per minute)
- [ ] How should the "anonymous alias" be generated? Random adjective+noun? Numeric suffix? User-selectable?
- [ ] What is the campus scoping model? Single campus, multi-campus, or global?
- [ ] Should the chat support emoji? Quick replies? Read receipts?
- [ ] What is the content moderation policy for the post-MVP phase? Which 微信内容安全 API endpoints should be used?

### 6.3 Deliberately Deferred

| Item | Why Deferred | When to Revisit |
|------|-------------|----------------|
| User authentication / real-name verification | Anonymous interaction is the core mechanism. Adding identity verification would contradict the walking skeleton's design principle. | Post-prototype, if user research shows that anonymity leads to trust/safety issues that outweigh the engagement benefit. |
| Content moderation (automated) | Not required to demonstrate the core flow. Moderation adds API calls, review queues, and policy decisions that slow down the skeleton. | Before any public distribution or app store review. Required for 小程序审核. |
| Push notifications (订阅消息) | In-app notifications are sufficient for the walking skeleton. Push adds template approval and user opt-in complexity. | Post-prototype, if user testing shows that match response rates drop significantly when users are not actively in the app. |
| Image/media uploads in posts | Adds cloud storage costs, moderation complexity, and a richer post composer UI. Text-only posts keep the skeleton focused on the matching mechanism. | Post-prototype, if user feedback indicates text-only posts limit engagement. |
| Algorithmic feed ranking | Chronological sort is simple, transparent, and sufficient to demonstrate the concept. Ranking algorithms require engagement data that doesn't exist yet. | Post-prototype, once there is enough usage data to train or tune a ranking model. |
| Multi-campus or global scoping | Single-campus assumption keeps the data model simple and the feed relevant. Scoping logic adds complexity to every query. | Post-prototype, if user research validates demand for cross-campus discovery. |
| Admin/reporting dashboard | No moderation means no review queue. Manual database queries are sufficient for prototype debugging. | When content moderation is implemented. |
